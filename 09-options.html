PART 2 · CHAPTER 09

9. 개발 범위 선택지 — 왜 풀스택은 아닌가, 왜 차트 모듈인가

2부의 척추 — 4개 옵션을 중립 비교하지 않고, '왜 C(차트 모듈)가 출발점인가'를 논증합니다.
⚡ 결론 (선결)
2026년에는 옵션 C — EyeChartPro CRM에 차트 모듈 확장이 6.4 의사결정 4축에서 4개 모두 일정 수준 충족하는 유일한 선택지입니다 — 단, ②실행 가능성과 ④리스크에는 아래 9.3에 명시된 단서가 동반됩니다.
옵션 A(풀스택)는 ②실행 가능성과 ④리스크에서 무너지고, 옵션 B(안과+외부청구 연동)는 ③ROI와 ②실행 가능성에서 약하며, 옵션 D(부가 솔루션)는 ①전략 적합도가 낮습니다. 단계 점프(C → 풀스택)는 차후 Stage 3에서 재평가합니다.

9.1 선택지 A: 풀스택 EMR 자체 개발 — 현시점 비현실적인 이유

'풀스택 자체 EMR'은 접수·예약·진료·처방·청구·수납·통계까지 전 영역을 자체 개발하는 옵션입니다. 6.4 4축으로 평가하면 다음과 같습니다.

평가 축판정근거
① 전략 적합도○ 높음모든 자산이 한 시스템에 정렬됨. 데이터 사이클이 사내에서 닫힘
② 실행 가능성✕ 무너짐SOFTCRM 현대화(2026 치환→2027 PG 이전), EyeChartPro, LOOCUS, OMS, VXV 등 확정 우선순위만으로도 인력 한계. 청구·DUR·진료정보교류 도메인 인력 부재. 한 해 안에 추진 불가
③ ROI△ 보류장기 가치는 가장 큼. 단 정량 산정은 Stage 2 후 시점에서 의미가 있음
④ 리스크✕ 무너짐청구 사고·진료 중단·환자안전 사고가 동시에 발생할 표면적이 가장 큼. 6~12개월 인증 + 베타 운영 중에도 진료를 멈출 수 없음 → 폴백 부담 최대
🛑
판정: 옵션 A는 2026년 추진 불가. 단, '영원히 안 한다'가 아니라 Stage 3(2029~) 시점에 시장·여력·도메인 조건을 재평가하여 GO/NO-GO를 결정하는 옵션으로 보류합니다(10장 참조).

9.2 선택지 B: 안과 특화 EMR + 외부 청구 모듈 연동

차트·진료·검사·수술기록 등 안과 도메인 영역만 자체 개발하고, 청구·DUR·진료정보교류는 외부 EMR 모듈과 연동하는 옵션입니다.

평가 축판정근거
① 전략 적합도○ 높음차트·검사·영상이 사내 자산. 데이터 사이클이 닫힘
② 실행 가능성△ 약함외부 청구 모듈 연동 계약 + 인증 책임 분담 + 데이터 동기화 설계가 누적. 청구 사고 발생 시 책임 소재 분쟁 리스크
③ ROI△ 약함외부 청구 모듈 라이선스 비용 + 자체 차트 개발 비용을 모두 부담. 비용 회피 효과 크지 않음
④ 리스크△ 중간청구 영역은 외부에 위임 가능. 단 두 시스템 간 데이터 동기화 실패 시 의료사고 가능성

판정: 옵션 B는 자산 정합성은 좋지만 '어차피 차트를 자체 개발한다면 그 시작은 EyeChartPro CRM 위에서가 더 유리' → 옵션 C가 더 빠르고 안전한 출발점입니다. B는 C에서 차트 모듈이 자리잡은 후에 청구 외부 위임을 옵션으로 검토할 수 있습니다.

9.3 선택지 C: EyeChartPro CRM에 차트 모듈 확장 (권고안)

EyeChartPro CRM(예약·상담·재방문 관리)에 차트 영역(SOAP·필차트·이미지·안과 전용 측정 필드)을 모듈로 추가하는 옵션입니다. 마스터 시스템 분담은 8·10장과 동일 — 환자·예약·청구는 외부 EMR이 마스터, 차트·필차트·안과 측정·영상은 신규 차트 모듈이 마스터. EyeChartPro CRM의 예약·상담은 외부 EMR과 양방향 동기화로 운영됩니다.

평가 축판정근거
① 전략 적합도○ 높음차트·검사·영상이 EyeChartPro에 적재됨 → LOOCUS AI와 직결. 데이터 사이클의 핵심 부분이 사내에서 닫힘
② 실행 가능성△ 통과 (단서 있음)풀스택 대비 신규 시스템 운영 부담은 제한적이고, 베타 의료기관도 사내 자산(B&VIIT). 단 차트·SOAP·필차트·측정값·영상 적재가 시작되는 순간 EyeChartPro는 단순 CRM이 아니라 의무기록성 데이터 운영 영역으로 들어옴 — 보안, 환자 동의 구조, 감사로그, 백업·보존 의무가 동반됨
③ ROI○ 통과데이터 자산화 + 안과 특화 UX + 카테고리 선점 1단계가 동시에 발생. 외부 EMR과 병행 운영이므로 비용 회피는 부분적
④ 리스크△ 통과 (단서 있음)청구 영역의 사고 리스크는 외부 EMR이 담당 → 우리에게 이전되지 않음. 차트 모듈 장애 시 외부 EMR로 폴백 가능 (8.4 참조). 단 의무기록 영역의 개인정보 유출·동의 미흡·감사로그 누락은 우리 측 리스크로 새로 발생 — 8.5 컴플라이언스 항목으로 동시 관리 필요
✅
옵션 C가 권고안인 이유
4축이 모두 일정 수준 충족(②④에는 단서 동반) + 안과 특화 EMR 카테고리 진입 1단계로서 의미가 명확. 신규 시스템을 만드는 것이 아니라 이미 운영 중인 EyeChartPro CRM이 본업의 절반(예약·상담)을 이미 처리하고 있다는 사실 위에서 차트 영역만 더하는 구조이므로 풀스택 대비 진입 비용이 낮음 — 단, 차트 영역이 들어오는 순간 의무기록성 데이터 운영 부담(보안·동의·감사로그)은 새롭게 발생함을 9.3 평가표에 명시했습니다.

9.4 선택지 D: 기존 EMR 연동형 부가 솔루션

외부 EMR 위에 'AI 보조 판독·SOAP 자동초안·검사 데이터 시각화' 등 부가 솔루션을 얹는 옵션입니다.

평가 축판정근거
① 전략 적합도✕ 낮음차트·검사 데이터는 여전히 외부 EMR에 잠재. 데이터 사이클이 사내에서 닫히지 않음 — 6.3의 핵심 가정 무너짐
② 실행 가능성○ 통과외부 EMR API/연동 표준에 의존하면 추진 가능. 인증 부담 가장 낮음
③ ROI△ 보통AI 부가 매출 가능. 단 데이터 자산화 효과 없음 → 장기 가치 약함
④ 리스크○ 낮음EMR 본체에 손대지 않으므로 진료 중단·청구 사고 리스크 낮음

판정: 옵션 D는 단기 매출/실행 가능성은 좋지만 '우리가 EMR을 검토하는 본질적 이유(데이터 사이클 내재화)'를 충족하지 못함. C와 별개로 단기 부가 사업으로 병행은 가능하나, '자체 EMR 검토'의 답으로는 부적절.

9.5 각 선택지에 AI-네이티브 레이어를 얹었을 때

LOOCUS AI는 어느 옵션과 결합해도 가치를 만들지만, SaMD(의료기기 SW) 인허가 분류에 따라 위험·비용이 달라집니다. 옵션별로 어디까지 안전하게 결합할 수 있는지 정리합니다.

⚠️ 임상책임·SaMD 리스크
AI 보조 판독·예측 기능을 의료 의사결정에 직접 연결할 경우 식약처 의료기기 SW(SaMD) 인허가 트랙으로 진입합니다. 이 경우 임상시험 + 인허가 비용 + 사후관리 책임이 EMR 사업과 동시에 부과되므로, 옵션 C 단계에서는 '비-의료기기 등급의 기록 보조 AI'에 집중하고, SaMD 등급은 Stage 3 이후로 분리합니다.
